1 — Why this matters — utility for Laurie

The core value isn't filing or finding estimates — Laurie already organizes Word docs by company and course name. The real value is keeping the estimate alive as a live calculator after the Word doc has been sent.

Once Laurie exports a .docx today, that estimate is frozen. Editing it means manually updating Word tables with no live math. The database means she can always come back to the working tool version — with all the live totals, checkboxes, and recalculation still intact. That's what she described as "editing the information online multiple different ways."

What the database actually unlocks

Client asks for a revision
Client says "can we cut the QA pass?" or "what if we add ADA?" Laurie reopens the saved estimate, checks or unchecks one row, and totals recalculate instantly. Without saving, she's either manually editing Word tables or rebuilding the whole estimate from scratch.
Live scenario testing
During a sales call: "what if we do Rise instead of Storyline?" or "what does the 45% margin look like?" She can open the estimate right there and toggle things in real time. The live tool is far more powerful than the static Word doc for this.
Duplicate for similar clients
A new client with nearly identical scope — compliance training, same team, similar length. Duplicate a saved estimate, update the company and course name, tweak a few hours, export a fresh .docx. Two minutes instead of twenty.
Mid-draft saves
Estimates sometimes take more than one sitting. Save mid-way, close the browser, come back and everything is exactly as she left it — no re-entry.
Honest time savings: This won't save 30–60 minutes per estimate on its own — the tool already handles that. The savings are situational: revision requests (10–20 min saved), duplicating for similar clients (15–25 min saved), scenario testing in a sales call (5–10 min saved). The bigger gain is qualitative — the estimate stays flexible instead of becoming a static document the moment it's exported.

2 — What's in scope (and what's not)

In scope

  • Supabase project setup, schema, and browser client
  • GitHub OAuth login (Laurie's existing GitHub account via her Supabase account)
  • Save estimate button on the Export Preview screen
  • Update saved estimate (if viewing a previously saved estimate)
  • My Estimates list screen — grouped by company, with Open / Duplicate / Delete actions
  • Restore full App state from a saved estimate
  • New Estimate button (clears state back to blank)
  • Unsaved-changes indicator (dot on the Save button)

Explicitly out of scope

  • Multi-user accounts or permissions
  • Sharing estimates with clients
  • PDF export or storing the .docx file — .docx is always regenerated client-side on demand
  • SOW generation, invoicing, or CRM integration
  • Editing or commenting on saved estimates from the list view (open → edit → save is the flow)

3 — Stack decisions

Supabase Chosen

Hosted Postgres with a browser-native JS client. Laurie already has an account linked to her GitHub. Free tier supports up to 500 MB storage and 2 GB bandwidth per month — more than sufficient for a single-user tool storing JSON blobs. No server required: the browser talks directly to Supabase via the @supabase/supabase-js client, keeping the architecture the same as today (fully client-side, no backend).

Auth method
GitHub OAuth  Laurie signs in with GitHub — one click, no password to remember. Supabase handles the OAuth flow and issues a session token stored in localStorage. On return visits the session auto-restores.
New package
@supabase/supabase-js — the official Supabase client. One install, no other dependencies. The existing docx and file-saver packages are untouched.

4 — Data model

One table. Estimates are stored as a JSON blob (full App state) plus a few quick-display columns so the list screen can render without parsing every blob.

ColumnTypeNotes
iduuid (PK)Auto-generated. Stable identifier — survives edits and re-saves.
user_iduuid (FK → auth.users)Supabase fills this automatically. Row-level security ensures Laurie only sees her own rows.
company_nametextCopied from the Company Name field at save time. Used for grouping in the list view.
course_nametextCopied from the Course Name field. Display label in the list.
categoriestext[]e.g. ['microvideo', 'rise360'] — for a quick visual badge in the list without parsing the blob.
total_internalnumericSnapshot of internal cost at save time. Shown in the list for quick reference.
total_clientnumericSnapshot of client price at save time.
margin_pctinteger40, 45, or 50. Shown as a badge in the list.
state_jsonjsonbFull serialized App state — all category tasks, hours, checked states, additionalMinutes, ADA toggles, module counts, additional videos. Everything needed to restore the estimator exactly.
created_attimestamptzAuto-set on insert.
updated_attimestamptzUpdated on every save. Shown in list as "Last saved: …"
Why JSON blob + display columns? The state structure evolves as new task types are added. Storing the full state as JSON means no schema migrations when the estimator gains new features. The display columns (company, total, margin) are only ever written, never queried — so they don't need to be perfectly normalized.

Row-level security (RLS)

Supabase's RLS policy is two lines of SQL: users can only SELECT, INSERT, UPDATE, and DELETE rows where user_id = auth.uid(). Since there's only one user right now this is mostly a safety net, but it's the correct pattern and costs nothing.

5 — Authentication

Supabase Auth with GitHub OAuth. Laurie's existing Supabase account (linked to GitHub) handles everything — no new credentials, no separate password.

Session behavior

  • On first visit: Login screen shown. Laurie clicks "Sign in with GitHub", gets redirected to GitHub, approves, and lands back on the estimator.
  • On return visits: Supabase auto-restores the session from localStorage. No login prompt unless the session has expired (default: 1 hour idle, extendable to 30 days).
  • Sign out: Available in the top-right corner of the app. Clears session and returns to Login screen.

What the app does before auth check

The estimator itself loads immediately — no blocking spinner. The auth check happens in the background. If the user is not signed in, a non-intrusive banner appears at the top: "Sign in to save your estimates". All estimation functionality works without being signed in; Save is the only gated action.

Action details

Open
Loads the saved state_json into App, navigates back to the main estimator screen. The save button will show "Update" since this is a known estimate (has an id).
Duplicate
Creates a new row in the database with the same state_json but a new id and created_at. Immediately opens it in the estimator. Useful for similar-client re-use.
Delete
Shows a one-line confirmation ("Delete Compliance Training 2026?"). On confirm, removes the row. No undo — the .docx export is the backup if Laurie ever needs recovery.
New Estimate
Clears all App state back to the blank starting point and navigates to the main estimator. If there are unsaved changes a prompt appears first: "You have unsaved changes — Save first or discard?"

7 — Full user flow

Flow A — Creating and saving a new estimate

  1. Laurie visits cc-estimator.vercel.app. Session auto-restores (no login prompt on return visits).
  2. She builds the estimate as today — selects categories, checks tasks, edits hours.
  3. Clicks "Export / Preview". The Export Preview screen opens.
  4. She clicks Save Estimate. A row is written to Supabase. Button becomes "✓ Saved".
  5. Optionally clicks Download .docx as well.

Flow B — Returning to a saved estimate

  1. Laurie clicks My Estimates in the header.
  2. Finds the estimate by company name. Clicks Open.
  3. App state is fully restored — all categories, checked tasks, hours, modules, ADA toggles exactly as she left them.
  4. She makes changes, clicks Export / Preview, clicks Update Estimate. Existing row is overwritten.

Flow C — Duplicating for a similar client

  1. Opens My Estimates. Finds an existing estimate for a similar client.
  2. Clicks Duplicate. A new estimate is created and opened immediately.
  3. Updates Company Name and Course Name at the top. Adjusts hours/tasks as needed.
  4. Saves as a new entry under the new company name.

Flow D — First-ever visit (login)

  1. Laurie visits the site for the first time after the database launch.
  2. A non-blocking banner: "Sign in to save your estimates — Sign in with GitHub".
  3. She clicks it. GitHub OAuth prompt. She approves. Returns to the estimator, now signed in.
  4. Session persists for 30 days of activity.
