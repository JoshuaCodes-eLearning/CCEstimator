Section 1

01Who uses this, and what changes for them

The estimator has one user: Laurie, a project manager at Cobblestone Consulting. She uses the tool to build eLearning quotes — choosing task categories, editing hours per person, and exporting a Word document for clients.

Right now, once she closes the browser, the estimate is gone. The Word file is the only record. Any revision — a client asking to cut a task, a margin question during a call, a new quote for a similar client — means starting from scratch or manually editing Word tables by hand.

This feature changes that. Every estimate can be saved as a live working version: something she can come back to, adjust, and re-export at any time.

What Laurie can do today
  • Build an estimate with live math
  • Export a formatted Word document
  • That's it — everything resets when she closes the tab
What she'll be able to do
  • Save any estimate with one click
  • Browse all saved estimates in a searchable, sortable list
  • Reopen a past estimate and pick up exactly where she left off
  • Revise and re-export without rebuilding from scratch
  • Save a copy instead of overwriting an original

These choices have been made. Everything in sections 3–7 follows from them.

Section 4

04Save Estimate — new vs. existing logic

When Laurie clicks Save Estimate, the tool checks whether she's working on a brand new estimate or one she loaded from a previous save. The behavior is different in each case.

Brand new estimate

No saved record exists. The estimate is saved immediately — assigned a unique ID and written to the database. No dialog, no extra step. The button updates to confirm it was saved.

Loaded from a previous save

A record already exists. Before writing anything, the tool asks: overwrite the current version, or keep the original and create a separate copy?

The overwrite dialog

When a saved version already exists, clicking Save Estimate opens a small dialog before any data is written:

cc-estimator.vercel.app
┌───────────────────────────────────────────────┐ │ Save Estimate [✕] │ │ ───────────────────────────────────────── │ │ │ │ "Cobblestone — Compliance Training 2026" │ │ already has a saved version. │ │ │ │ ┌─────────────────────────────────────────┐ │ │ │ Overwrite current estimate │ │ │ │ Replaces the saved copy with your │ │ │ │ latest changes. │ │ │ └─────────────────────────────────────────┘ │ │ │ │ ┌─────────────────────────────────────────┐ │ │ │ Save as a separate estimate │ │ │ │ Keeps the original untouched and │ │ │ │ creates a new independent copy. │ │ │ └─────────────────────────────────────────┘ │ │ │ └───────────────────────────────────────────────┘
Why both options? Overwrite handles the common case: "I revised this estimate and want to replace it." Save as separate handles the other case: "I'm building a Phase 2 and need Phase 1 to stay intact" — or "I want to try a different scope without losing what I had."

What "save as separate" does

Creates a new estimate record with a new unique ID. The original is untouched. The new copy becomes the "active" estimate — so if Laurie saves again after more changes, the dialog will ask about the new copy, not the original.

Whether the new copy is auto-named (e.g. "Compliance Training 2026 — copy") or Laurie names it herself is an open question — see Section 8.

Section 6

06Loading an estimate — double-click to prefill

From the View Estimates modal, Laurie double-clicks any row. The modal closes, the estimator screen appears, and every field is fully restored to exactly how the estimate was when it was last saved.

1

Laurie double-clicks an estimate row in the modal.

2

The modal closes. The main estimator screen appears.

3

Everything is prefilled: selected categories, checked and unchecked tasks, all hours per person, additional minutes, ADA toggles, number of modules, additional video settings — exactly as last saved.

4

Laurie can edit anything freely. All live math recalculates as normal.

5

When she clicks Save Estimate, the tool recognizes this as an existing record and shows the overwrite-or-save-as dialog from Section 4.

The core guarantee: Loading an estimate is completely lossless. Every setting that was saved comes back exactly as it was. There should be no visible difference between an estimate built from scratch and one loaded from a save.
Section 7

07What gets stored per estimate

Each saved estimate holds two kinds of data: quick-display fields that power the list view, and the full estimate state that enables lossless reload.

FieldWhat it isUsed for
Unique ID Auto-generated on first save, never changes Identifies the estimate; drives the overwrite-vs-save-as check
Company name From the Company Name field at save time Shown in the list; searchable
Course name From the Course Name field at save time Shown in the list; searchable
Categories Which category blocks were active (Microvideo, Rise 360, Storyline 360) Shown as badges in the list
Client price The client-facing total at save time Shown in the list for at-a-glance reference
Margin % 40%, 45%, or 50% Shown in the list; filterable in a later iteration
Saved date When the estimate was last saved or updated Shown in the list; used for "Most Recent" sort
Full estimate state All tasks, all hours per person, checked/unchecked states, additional minutes, ADA toggles, module counts, additional video configurations — everything Restores the estimator exactly on load
The Word document is not stored. The .docx is always generated fresh from the live estimate state when Laurie clicks "Export to Word." Saving an estimate and exporting a Word file are separate, independent actions — she can do one, both, or neither in any order.
Section 8

08Open questions — to flesh out later

These items are intentionally unresolved. Nothing in sections 1–7 is blocked by any of them — they'll be defined before the relevant part of the feature is built.

To define
Where does View Estimates live? Confirmed on the Export Preview action row. Does it also appear somewhere on the main estimator — in the header, as a top-level button? Or is it only accessible through the Export Preview screen?
To define
Full search and filter variables. Company name, course name, and date are confirmed. What else — filter by category type, by margin %, by date range? To be finalized before the View Estimates modal is built.
To define
Full sort options. Most Recent and Alphabetical by Company are confirmed. Anything else — sort by client price, sort by course name, sort by date created vs. date last updated?
To define
Delete from the list. Can Laurie delete a saved estimate from within the View Estimates modal? If yes, does it require a confirmation step? This wasn't addressed in the initial brief.
To define
Naming a "save as separate" copy. When Laurie saves as a separate estimate, does it get an automatic name ("Compliance Training 2026 — copy") or does she get a chance to give it a new name before it saves?
To define
Unsaved changes guard when opening View Estimates. If Laurie is mid-estimate and opens View Estimates, then double-clicks a different one — what happens to the in-progress work? Should there be a prompt: "You have unsaved changes — save first or discard?"
To define
Authentication approach. Saving to a database requires some form of login to keep Laurie's estimates private and persistent across devices/sessions. Auth method and flow are specified in DatabaseHLDD.html — confirm that approach still fits before building.
